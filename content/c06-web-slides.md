# 网页就是幻灯片：告别 PPT 的 Slidev 路线

## 为什么不用 PowerPoint / Google Slides

课堂笔记里那行「HTML ↔ Microsoft PowerPoint, Google Slides」说的就是这个选择题。结论：**当制作者是 AI 时，HTML 是碾压性的选择。**

| | HTML（本课路线） | PowerPoint | Google Slides |
|---|---|---|---|
| AI 能否直接生成 | ✅ 就是文本文件，AI 的主场 | 勉强（专用库，效果差） | 差（API 不支持自定义图形和字体） |
| 美观上限 | 无上限（逐像素控制） | 中 | 低 |
| 数学公式 | ✅ KaTeX 完美渲染 | 麻烦 | 麻烦 |
| 分发 | 发一个链接 | 传文件，字体易变形 | 发链接 |
| 别人可编辑 | ❌ | ✅ | ✅ |

PPT 家族唯一的真优势是「别人能改」。如果你不需要——个人展示、课堂 pre、论文汇报——就没有理由忍受它的天花板。

## Slidev：给 HTML 幻灯片一个成熟的壳

裸写 HTML 也行，但 [Slidev](https://sli.dev/) 白送太多东西：**Markdown 写内容、CSS 管样式、演讲者模式（备注+计时+下一页预览）、PDF/PNG 导出、一键构建成静态网站**。今天两个 deck 都是这条路线：

```bash
npm install @slidev/cli @slidev/theme-default   # 装（版本需 ≥ 52.17.0）
npx slidev                                       # 本地写作预览
npx slidev build --base /仓库名/ --out docs      # 构建，产物进 docs/
# 然后 git push + 开 Pages（/docs 目录）即上线
```

`--base /仓库名/` 就是上一章说的子路径前缀，**项目站必须加**，忘了就全白屏。

## 今天真实踩过的四个坑（每个都消耗了真实时间）

1. **KaTeX 公式在 HTML 块里不渲染**。Slidev 的 Markdown 里如果用 `<div>` 排版，块内的 `$$公式$$` 会原样显示。解法：公式前后留空行，让它回到 Markdown 上下文；图注里的下标改用 HTML `<sub>` 标签。
2. **图片放 `public/` 用绝对路径会构建报错**（被当成模块导入）。解法：图片放项目内目录，用相对路径 `./images/xxx.png`，构建工具自动处理哈希和 base 前缀。
3. **本地预览服务器不支持路由**：直接访问 `.../5` 这种深链接 404，要从入口页进去再翻页。线上没这个问题（构建产物带 404 回退）。
4. **截图截到过渡动画**：自动化检查时，翻页后要等 1–2 秒再截图，否则截到两页叠影，误判为排版事故。

## 质量纪律（来自另一个 slides 项目的血泪经验）

这些规则来自更早的一个演示文稿项目，已被反复验证：

- **硬字数预算 + 装不下就拆页**。绝不用「自动缩小字号」凑合——页与页字号忽大忽小，一眼机器味。
- **写完必须截图检查，且要派一个独立的 AI 审阅**。写代码的 AI 自己看自己，会「看到自以为写了的东西」，对着有 bug 的页面自我祝贺。独立审阅的开场白要写：「假设这里面有问题——把它们找出来。」
- **给审阅者加反向约束**：禁止为了通过检查而删设计、简化排版。有真实案例是 AI 为了消除「内容溢出」警告，把版面一点点啃光，40 轮不收敛。
- **标题写结论，不写主题**。「法英两国控制了非洲三分之二的土地」远胜「殖民地分布」。听众扫一眼标题就能带走信息。
- **图表配色要过校验**（对比度 ≥3:1、色盲可区分），别凭感觉挑颜色。
- **中文字体**：Slidev 默认从 Google Fonts 拉字体，国内观众打开会退化成系统字体。要么锁定系统字体栈（`fonts: provider: none`），要么本地打包字体文件。

## 从一篇论文到一个 deck 的实操流程

今天把一篇高中数学论文（LaTeX 源码 + 23 张图）变成了 14 页幻灯片，流程可复制：

1. 把论文文件夹路径给 Claude Code：「读一读，研究一下」；
2. 交代保真要求：「尽可能多地保留原文的关键内容、概念、图片、引用」；
3. 交代风格：「红白黑配色，无衬线字体，瑞士设计」；
4. AI 产出 → 截图检查 → 部署 → 你线上翻一遍做最终验收。
