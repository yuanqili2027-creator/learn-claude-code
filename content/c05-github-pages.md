# GitHub Pages：把文件夹变成全世界可访问的网站

## 原理一句话

GitHub Pages = GitHub 免费帮你把仓库里的 HTML 文件**原样挂到互联网上**。没有服务器要买，没有域名要备案，推送后约 30–60 秒生效。

先认识三个部署平台（课堂笔记原文）：

| 平台 | 定位 | 适合 |
|------|------|------|
| **GitHub Pages** | 代码分享平台附带的静态托管 | 个人主页、文档、幻灯片（本课路线）|
| **Vercel** | 全栈网站开发平台 | 需要后端逻辑的动态网站 |
| **Cloudflare** | 网络分发和安全平台 | 需要访问控制、国内访问优化 |

新手先用 GitHub Pages 就够了——它和你的代码仓库天然一体。

## 两种站点，网址不同

| 类型 | 仓库名 | 网址 | 用途 |
|------|--------|------|------|
| **用户主站** | 必须叫 `用户名.github.io` | `https://用户名.github.io/` | 个人主页，一人一个 |
| **项目站** | 任意名字 | `https://用户名.github.io/仓库名/` | 每个项目一个，数量不限 |

今天实战里：个人主页用了用户主站；两个幻灯片各占一个项目站。**推荐这种「一个作品一个仓库」的组织方式**，互不干扰，删改自由。

## 开启方式一：网页操作（新手友好）

仓库页面 → **Settings** → 左栏 **Pages** → Source 选 `Deploy from a branch` → Branch 选 `main`、目录选 `/ (root)` 或 `/docs` → Save。等一分钟，页面顶部会显示你的网址。

## 开启方式二：命令行（Claude Code 用的就是这个）

```bash
gh api repos/用户名/仓库名/pages -X POST \
  -f "source[branch]=main" -f "source[path]=/"
```

解释：`gh api` 是直接调用 GitHub 的开放接口（API）；`-X POST` 表示「创建」；两个 `-f` 参数指定从哪个分支、哪个目录发布。用户主站（`用户名.github.io` 仓库）通常推送后自动开启，这条命令会返回「已存在」，无妨。

## 根目录还是 /docs？

- **纯手写的网站**（如个人主页）：文件就在根目录，选 `/ (root)`。
- **需要构建的项目**（如 Slidev 幻灯片）：源码在根目录，构建产物输出到 `docs/`，Pages 选 `/docs`。这样一个仓库同时保存「原料」和「成品」。

## 验证是否上线

```bash
curl -s -o /dev/null -w "%{http_code}" https://用户名.github.io/仓库名/
```

`curl` 是命令行浏览器；这条命令只取回**状态码**：`200` = 正常上线，`404` = 还没生效（等一会）或路径错误。今天的实战里，每次发布后 Claude Code 都用它自动轮询确认，而不是口头说「应该好了」。

## 三个今天踩过 / 绕过的坑

1. **子路径问题**：项目站的网址带 `/仓库名/` 前缀，网页里所有资源路径必须考虑这个前缀（构建工具一般有 `--base` 参数）。忘了就是「页面全白、资源 404」。
2. **Pages 是公开的**：免费版 Pages **没有访问控制**，私有仓库发的 Pages 站点依然全网可见。别把不想公开的东西（尤其是密钥）放上去。
3. **发布有延迟**：推送后 30–60 秒才生效。改完看不到变化，先强制刷新（`⌘⇧R`），再怀疑人生。

## 对 Claude Code 说的话

> 「帮我把这个文件夹发布成 GitHub Pages 网站，建一个叫 ___ 的公开仓库，发布后验证能正常访问，把网址给我。」

一句话包含了：动作（发布）、仓库名（你定）、验收（验证 + 给网址）。这就是全部。
